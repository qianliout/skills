# Project Planner

将复杂项目拆解为可执行的任务，含时间线、依赖关系、里程碑和风险清单。用于项目范围定义、工作拆解、依赖分析、时间估算、资源分配和风险评估。

## When To Load

- 定义项目范围和交付物
- 创建工作拆解结构（WBS）
- 识别任务依赖关系
- 估算时间线和人力投入
- 规划里程碑和阶段
- 分配资源
- 风险评估和缓解

## 规划流程

### 1. 定义成功标准
- 最终目标是什么？
- 成功标准是什么？
- 什么定义"完成"？
- 约束条件是什么（时间、预算、资源）？

### 2. 识别交付物
- 主要输出是什么？
- 哪些里程碑标记进度？
- 有哪些依赖关系？
- 哪些可以并行？

### 3. 拆解任务
- 每个任务：2-8 小时工作量
- 明确的"完成"标准
- 可分配给单个人
- 可测试/可验证的完成

### 4. 映射依赖
- 哪些必须先做？
- 哪些可以并行？
- 哪些在关键路径上？
- 瓶颈在哪里？

### 5. 估算和缓冲
- 最好情况、最可能情况、最坏情况
- 为未知增加 20-30% 缓冲
- 考虑评审/测试时间
- 包含风险应急时间

### 6. 分配和跟踪
- 谁负责每个任务？
- 需要什么技能？
- 如何跟踪进度？
- 何时进行检查？

## 任务尺寸指南

| 分类 | 特征 | 处理方式 |
| ---- | ---- | -------- |
| **过大** (>2 天) | 难以准确估算、进度难追踪、阻塞其他工作 | 拆分为子任务 |
| **合适** (2-8 小时) | 明确交付物、一人可完成、进度每日可见 | 直接使用 |
| **过小** (<1 小时) | 可能过度规划、开销太大 | 合并相关微任务 |

## 估算技巧

### 三点估算

```
乐观 (O): 最佳情况
最可能 (M): 预期情况
悲观 (P): 最坏情况

期望值 = (O + 4M + P) / 6
```

### T-Shirt 尺寸

- **XS**: < 2 小时
- **S**: 2-4 小时
- **M**: 4-8 小时（1 天）
- **L**: 2-3 天
- **XL**: 1 周

*超过 XL 的任务必须继续拆分*

## 示例输出

```markdown
## 项目：公司网站改版

**时间线**：6 周
**团队**：1 设计师、2 开发者
**约束**：必须保持所有现有功能，无新工具预算

## 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 设计反馈延迟 | 高 | 中 | 提前安排评审，限制修改轮次为 2 |
| 范围蔓延 | 高 | 高 | 第 1 周后锁定需求，新需求记录为 Phase 2 |
| 内容未就绪 | 中 | 中 | 提前开始内容迁移（第 4 周），必要时使用占位符 |
| 技术问题 | 中 | 低 | 第 5-6 周留缓冲，准备备用托管方案 |
| 团队成员请假 | 中 | 低 | 开发者交叉培训，设计师必要时可做基础 HTML/CSS |

- 所有现有功能保持正常
```
