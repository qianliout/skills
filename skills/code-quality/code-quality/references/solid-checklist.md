# SOLID Checklist

在代码审查明确涉及架构和职责分层时使用。

## SRP

- 一个模块是否同时承担 HTTP 适配、业务编排、持久化或展示等多种职责。
- 一个类型是否有多个独立变更原因。
- 一个函数是否在做多段彼此独立的工作流。

## OCP

- 新增一种行为时，是否必须修改核心分支逻辑，而不是通过扩展点扩展。
- 是否因为缺少策略、接口、hook 或配置边界，导致需求增长只能继续堆叠 `if` 或 `switch`。

## LSP

- 子类型是否破坏父类型承诺，要求调用方增加类型判断。
- 重写后的行为是否弱化了原本保证，或者引入 no-op、异常路径或特判。

## ISP

- 接口是否过宽，导致实现方被迫声明无用方法。
- 调用方是否只需要少量能力，却依赖了一个臃肿接口。

## DIP

- 高层业务是否直接依赖数据库、网络、文件系统等具体实现。
- 是否存在硬编码依赖，导致实现难以替换、测试难以隔离。

## Common Smells

- 过长函数和深层嵌套。
- Data clumps、primitive obsession、shotgun surgery。
- 死代码、推测式抽象、魔法数字和魔法字符串。

## Review Heuristics

- 优先按职责拆分，而不是按文件大小拆分。
- 只有第二个真实场景出现时，再考虑提取抽象。
- 建议重构时先强调最小安全拆分路径，避免一次性重写。
