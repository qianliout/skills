# Code Review Expert

用于审查当前代码改动，重点关注正确性、安全性、可维护性、架构分层和遗漏测试。默认执行只读审查，除非用户明确要求实现修复。

## Workflow

1. 先用 `git status -sb`、`git diff --stat`、`git diff` 确认审查范围。
1. 识别关键路径：入口函数、状态写入、鉴权链路、外部调用、并发路径、测试覆盖和配置变更。
1. 如需特定维度，再补充读取对应 checklist reference，而不是一开始展开所有检查表。
1. 输出 findings 时按严重程度排序，优先说明会导致错误行为、回归或安全风险的问题。
1. 用户没有要求修复时，不直接改代码。

## Severity Levels

- `P0`: 安全漏洞、数据损坏、明显正确性错误、会阻断发布的问题。
- `P1`: 高概率逻辑错误、严重设计缺陷、显著性能或稳定性回归。
- `P2`: 可维护性问题、中等风险代码味道、测试缺口、非阻断性行为风险。
- `P3`: 风格、命名、轻微结构建议。

## Review Focus

- 正确性：需求是否满足，边界条件是否遗漏，是否存在回归。
- 设计：职责是否清晰，依赖方向是否合理，新增抽象是否必要。
- 稳定性：错误处理是否完整，是否存在静默失败、竞态、资源泄漏。
- 安全性：输入输出安全、鉴权、敏感信息、默认配置是否安全。
- 测试：关键路径、失败路径和回归场景是否有覆盖。

## Output Format

```markdown
## Findings

1. P1 - `path/to/file`
   - 问题是什么
   - 为什么重要
   - 建议如何修复

## Open Questions

- 需要确认的需求或上下文

## Summary

- 总体结论
- 残余风险或测试缺口
```

## Rules

- 如果没有发现问题，要明确说明已检查的范围和未覆盖的风险面。
- 对大改动优先按模块分组审查，不按文件顺序机械罗列。
- 只有在问题真实影响行为、风险或长期维护成本时，才把它作为 finding。
- 审查引用代码位置时，必须给出具体文件或代码块范围。
